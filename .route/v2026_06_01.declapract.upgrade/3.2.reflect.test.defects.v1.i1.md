## defect 2: dpdm third-party cycles

**defect:** test:lint:cycles fails on node_modules cycles

**classification:** practice bug

**reasoning:**

searched practice:
```sh
gh search code --repo ehmpathy/declapract-typescript-ehmpathy "exclude"
```
result: `src/practices/lint/best-practice/.dpdmrc.yaml` contains:
```yaml
# only exclude devDependencies here
# if you exclude a prod dependency, you ship cycles to consumers and break their builds
exclude: []
```

**analysis:**

the comment refers to excluding specific packages from cycle detection in OUR code. but the actual issue is different: dpdm follows import chains INTO node_modules and detects cycles INSIDE third-party packages (bottleneck, @aws-crypto, @aws-sdk).

these cycles are:
- inside third-party code, not our code
- not under our control
- not something we ship to consumers (node_modules is not published)

**recommendation:** the practice template `.dpdmrc.yaml` should document how to exclude specific packages with internal cycles, not blanket exclude all node_modules.

example for this repo:
```yaml
exclude:
  - bottleneck
  - '@aws-crypto'
  - '@aws-sdk'
```

**why not exclude all node_modules:** blanket exclusion hides cycles in dependencies we control or should know about. only exclude specific third-party packages with known internal cycles.

**action:** file issue on ehmpathy/declapract-typescript-ehmpathy - add guidance for per-package exclusion

## defect 3: depcheck false positives

**defect:** test:lint:deps reports unused devDependencies (rhachet-roles-rhachet, test-fns)

**classification:** repo quirk

**reasoning:**

the packages reported as unused are:
- `rhachet-roles-rhachet` - dynamically loaded by rhachet framework
- `test-fns` - imported via jest config setup

these are repo-specific tooling choices. other repos may not use rhachet or may use different test utilities. the fix (add to .depcheckrc.yml ignores) is correct for this repo but not a universal pattern.

**action:** none - correct fix already applied

## defect 4: provision type error

**defect:** TypeScript error TS2339 on pkg.private access

**classification:** practice bug

**reasoning:**

searched practice:
```sh
gh search code --repo ehmpathy/declapract-typescript-ehmpathy "pkg.private"
```
result:
```
provision/github/declastruct.resources.ts: visibility: pkg.private === true ? 'private' : 'public',
provision/github/declastruct.resources.ts: private: pkg.private ?? false, // todo: why do we have to specify this twice?
src/practices/provision-github/best-practice/provision/github.repo/resources.ts: visibility: pkg.private === true ? 'private' : 'public',
src/practices/provision-github/best-practice/provision/github.repo/resources.ts: private: pkg.private ?? false, // todo: why do we have to specify this twice?
```

**analysis:**

the practice template directly accesses `pkg.private` without the case where the field doesn't exist in the type. TypeScript infers the type from the actual package.json contents, which may or may not include a `private` field.

the fix applied in this repo:
```typescript
const pkgPrivate = (pkg as unknown as { private?: boolean }).private;
```

this should be part of the practice template itself.

**action:** file issue on ehmpathy/declapract-typescript-ehmpathy
