these were genuine correctness defects the earlier suite shipped green. each fix carries a
regression test (rule.require.test-covered-repairs).

- **i011-b1 (no-expiry entry read as absent) — FIXED (Fix A):** a `{ expiration: null }`
  entry stores `expiresAtMse = Infinity`, which `JSON.stringify` writes as `null`. the old
  `parseEnvelopeExpiresAtMse` folded a non-number to `0` ("expired at epoch 0"), so every
  conditional op treated a permanent key as absent → a put-if-absent would **clobber** it. fix:
  `parseEnvelopeExpiresAtMse` now returns `number | null`, which keeps `null` = no-expiry;
  `readSourceEntry`'s `expiresAtMse` is `number | null`; `isRecordExpired`'s param type corrected
  from the `number | number` typo to `number | null`. **tests:** "no-expiry key reports a token"
  + "held no-expiry key must not be clobbered by put-if-absent".
- **i011-b2 (put-if-absent fast path bypassed the per-key lock) — FIXED (Fix B):** the old
  unlocked `O_EXCL` fast path could race a plain `set` (which now holds the lock) → lost update.
  fix: `saveToLocalConditional` collapsed to a single lock-covered read-check-write (dropped the
  unlocked fast path), so every writer to a key honors the one lock. **covered by** the extant
  put-if-absent held-key + expired-reclaim tests, now all under the lock.
- **i010-b1 (conditional get passed its precondition then yielded undefined) — FIXED (Fix C):**
  the post-check value fetch consulted the valid_keys index, so a physically-present but
