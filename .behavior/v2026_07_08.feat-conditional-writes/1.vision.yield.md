# 1.vision — conditional writes for simple-on-disk-cache

> wish: execute radio task #43 — implement `WithCacheConditionals` (put-if-absent + compare-and-set)

---

## the outcome world

### day-in-the-life

today, `simple-on-disk-cache` is last-writer-wins. every `set` clobbers whatever was
there. two processes that race to compute-and-store the same key both compute, both write,
and the loser's work is wasted. worse — two parallel `set`s to *different* keys can clobber
the shared `valid_keys` file and silently drop one of the keys (a known, documented race at
`src/cache.ts:358-373`).

after this wish, the cache gains a **version token** per key and three conditional moves:

- **put-if-absent** — "write only if nobody has this key yet." the first misser wins; the
  rest are told `SimpleCacheConditionError` and can converge on the winner's value.
- **compare-and-set** — "write only if the stored version still matches the one i read."
  a slow writer can no longer clobber a fresher value.
- **compare-and-delete** — "delete only my version" (a `set(key, undefined, { condition })`).
  this is the release half of a distributed lock.

### before / after

| scenario | before (1.7.11) | after (this wish) |
|----------|-----------------|-------------------|
| two processes miss the same key | both compute + both write (wasted work, last wins) | first `put-if-absent` wins; others catch `SimpleCacheConditionError` and re-read |
| slow writer races a fresh write | slow writer clobbers the fresh value | `compare-and-set` throws; slow writer knows to re-read |
| read the current token for a key | not possible | `await cache.version(key)` |
| build a distributed lock over the cache | best-effort (write → wait → re-read) | atomic: acquire = put-if-absent, release = compare-and-delete |
| require a conditionals-capable cache in a type | not expressible | `WithCacheConditionals<SimpleCacheAsync<string>>` |

### the "aha" moment

the value clicks when a consumer writes:

```ts
// with-simple-mutex, over an on-disk cache
const acquire = async (lockKey: string) => {
  try {
    await cache.set(lockKey, holderId, { condition: { version: null } }); // put-if-absent
    return { acquired: true };
  } catch (error) {
    if (error instanceof SimpleCacheConditionError) return { acquired: false }; // someone holds it
    throw error;
  }
};
```

one primitive — a version-guarded `set` — and suddenly the cache is a lock, a
compute-once gate, and an optimistic-concurrency store. no new surface area to learn; it's
the same `get`/`set` with one extra option.

## user experience

### usecases & goals

1. **stampede / compute-once** — only the first misser computes and stores; the rest
   converge. goal: never compute the same expensive thing twice in parallel.
2. **the valid-keys race** — the internal `valid_keys` writer can use compare-and-set to
   merge losslessly instead of read-before-write clobber. goal: never silently drop a key.
3. **optimistic concurrency** — a writer that read version `V1` writes with
   `condition: { version: 'V1' }`; if someone else wrote `V2` in between, the write throws.
   goal: no lost updates.
4. **distributed locks** — `with-simple-mutex` over any tier: disk = per-machine lock,
   s3 = global lock. goal: one atomic primitive that composes into mutexes.

### contract inputs & outputs

conforms **exactly** to the shipped `with-simple-cache@0.17.0` vocabulary
(`src/domain.objects/SimpleCache.ts`), so a consumer can `require WithCacheConditionals`.

**1. version reader (new method):**

```ts
version: (key: string) => Promise<string | undefined>;
// the current opaque version token for a key, or undefined when absent.
// equality-only — never parse or order it.
```

**2. get widened with an optional condition:**

```ts
get: (
  key: string,
  options?: {
    consistency?: SimpleOnDiskCacheConsistency;   // extant (from #40)
    condition?: { version: string | null };        // new
  },
) => Promise<string | undefined>;
// on a version mismatch → throw SimpleCacheConditionError
```

**3. set widened with an optional condition, ordered after expiration:**

```ts
set: (
  key: string,
  value: string | undefined | Promise<string | undefined>,
  options?: {
    expiration?: IsoDuration | null;                // extant
    condition?: { version: string | null };         // new, after expiration
  },
) => Promise<void>;
// condition.version === null   → put-if-absent   (write only if absent)
// condition.version === token  → compare-and-set  (write only if stored version matches)
// on a precondition miss → throw SimpleCacheConditionError
```

**error shape** (mirror `with-simple-cache`'s `SimpleCacheConditionError extends ConstraintError`):

```ts
new SimpleCacheConditionError('cache condition failed: version mismatch', {
  key,
  condition,          // { version: string | null }
  found,              // string | undefined — the current token
});
```

### what it looks like to leverage them

```ts
// compute-once (stampede protection)
const token = await cache.version(key);            // undefined if absent
if (token === undefined) {
  try {
    await cache.set(key, await computeExpensive(), { condition: { version: null } });
  } catch (e) {
    if (!(e instanceof SimpleCacheConditionError)) throw e;
    // someone beat us — fall through and read their value
  }
}
return cache.get(key);
```

```ts
// optimistic concurrency
const token = await cache.version(key);
const current = await cache.get(key);
const next = mutate(current);
await cache.set(key, next, { condition: { version: token } }); // throws if token went stale
```

### timelines

**put-if-absent, two racers on an open key:**

```
t0  A: version(key) → undefined
t0  B: version(key) → undefined
t1  A: set(key, a, { condition: { version: null } }) → OK   (mints token vA)
t1  B: set(key, b, { condition: { version: null } }) → THROWS SimpleCacheConditionError
t2  B: get(key) → a                                          (converges on winner)
```

**compare-and-set, stale token:**

```
t0  A: version(key) → vA
t1  C: set(key, c) → OK                                      (mints vC, unconditional)
t2  A: set(key, a2, { condition: { version: 'vA' } }) → THROWS (found vC, not vA)
```

## mental model

### how a user describes it to a friend

> "it's like a cache with an ETag. every value has a version stamp. i can say 'only write
> this if the version is still what i last saw' or 'only write this if it doesn't exist
> yet.' if i lose the race, it tells me instead of silently stomping."

### analogies & metaphors

- **ETag / If-Match** (http) — the exact same optimistic-concurrency model; s3 literally
  implements it with `If-None-Match` / `If-Match` headers.
- **CAS register** (concurrency) — compare-and-swap, the classic lock-free primitive.
- **`INSERT` that skips on conflict** vs **`UPDATE ... WHERE version = ?`** (sql) —
  put-if-absent and compare-and-set, respectively.
- **claiming a seat** — put-if-absent is "sit only if the seat is empty"; the version is
  the "who's sitting here" tag.

### terms: theirs vs ours

| they say | we say (in code) |
|----------|------------------|
| "version stamp" / "ETag" | `version` token (opaque string) |
| "write if empty" | `condition: { version: null }` (put-if-absent) |
| "write if unchanged" | `condition: { version: token }` (compare-and-set) |
| "it rejected my write" | `throw SimpleCacheConditionError` |
| "the lock" | put-if-absent + compare-and-delete |

we deliberately reuse `with-simple-cache`'s canonical names (`version`, `condition`,
`SimpleCacheCondition`, `SimpleCacheConditionError`) — no synonyms, no drift.

## evaluation

### how well it solves the goals

- **compute-once** ✅ — put-if-absent gives exactly-one-winner semantics.
- **valid-keys race** ✅ (enabled) — compare-and-set makes a lossless merge possible; note
  this is an *internal* payoff we can adopt, not a hard acceptance criterion of the wish.
- **optimistic concurrency** ✅ — version-guarded set.
- **distributed locks** ✅ (disk = per-machine; s3 = global — the s3 primitive **shipped**
  in `sdk-aws-s3@0.2.0`; this repo holds `0.1.1`, so a dep upgrade is required).
- **type conformance** ✅ — exported cache satisfies `WithCacheConditionals<SimpleCacheAsync<string>>`.

### pros

- **one primitive, many wins** — locks, compute-once, optimistic concurrency all fall out
  of a version-guarded set.
- **zero breaking change** — `condition` is optional; every extant `get`/`set`/`keys`
  call keeps working. `version` is additive.
- **contract-first** — conforms to a shipped, canonical vocabulary; consumers can depend on
  the type today.
- **backend-symmetric** — the cache's `condition: { version }` threads to disk (flock /
  O_EXCL) and maps to the cloud's `condition: { etag }` (`sdk-aws-s3@0.2.0`), where the etag
  **is** the version token.

### cons / tradeoffs

- **disk atomicity is per-machine only** — flock/O_EXCL are reliable on a local fs, but
  advisory-lock semantics degrade on NFS / mounted dirs. the task explicitly scopes disk as
  the per-machine tier, so this is acceptable, but it must be documented loudly.
- **cloud needs a dep upgrade** — s3 atomicity is now available (`sdk-aws-s3@0.2.0` shipped
  conditional writes), but this repo holds `sdk-aws-s3@0.1.1` (plain get/set/del). the cloud
  path requires a bump to `0.2.0` and a map of the cache condition onto the sdk's shape (see
  cloud-map note below). the `#5` issue tracker still shows OPEN, but the code merged and
  published — verified against `origin/main` + npm.
- **hash cost per op, but no format change** — with content-hash tokens, the version is
  **computed** from the value (via `hash-fns`), not persisted; so the JSON envelope is
  **unchanged** and legacy files stay readable with zero migration. the cost is a cheap hash
  per conditional op / `version(key)` call, over a value already in memory — negligible.

### edgecases & pit-of-success

| edgecase | how the contract keeps users safe |
|----------|-----------------------------------|
| put-if-absent on a **physically-present-but-expired** key | treat expired as logically absent → the write must **succeed**. requires read-check-under-lock, not a bare `O_EXCL` (which only sees physical presence). |
| CAS against an **absent** key (`version: token`, key gone) | `found === undefined !== token` → throws (correct: the thing you meant to update vanished). |
| `version(key)` on an **expired** key | returns `undefined` (logically absent), same as never-cached. |
| conditional **delete** (`set(key, undefined, { condition })`) | gate first, then delete → compare-and-delete = lock release. |
| condition supplied to a **memory-first** cache | the version lives in the source store; conditional ops must look the token up **source-first** and never trust the in-memory copy, then refresh memory after the write. |
| unconditional `set` (no `condition`) | unchanged last-writer-wins behavior; still mints a fresh token so a later CAS has something to compare. |
| **prior versionless entry** (a value written before this feature, no token in its envelope) | must count as **logically present**, so `found` is non-undefined for it. otherwise a put-if-absent (`version: null`) would see "no token → absent" and **clobber real data on upgrade**. with content-hash tokens this is clean: `found` is the **hash of the stored value**, computable even for a legacy envelope that lacks a persisted token — so a legacy entry naturally yields a real token, never `undefined`. |

## open questions & assumptions

### assumptions made

1. **version token = an etag (content hash), stored in the JSON envelope.**
   **confirmed by the wisher on 2026-07-08 ("etag is better").** the disk token is the hash of
   the stored value (via the extant `hash-fns` dep, `package.json` L72) — a mirror of s3, where
   the etag **is** the content hash. this makes both tiers share one model: *the token is the
   etag of the content*. it is opaque + equality-only, so it satisfies the `with-simple-cache`
   contract. supersedes the earlier "random per-write uuid" lean.
   - **one behavioral consequence (intended):** two writes of byte-identical content yield the
     **same** etag, so a compare-and-set against that etag treats identical content as
     "no change." that is exactly s3's behavior — a feature (idempotent content = idempotent
     version), not a surprise. lock/mutex holders write distinct holder ids, so their content
     (and etag) differ regardless.
2. **disk = per-machine tier.** cross-machine atomicity on mounted dirs is out of scope; the
   s3 tier is the global tier. (stated in the task's `.how per backend`.)
3. **expired == absent** for precondition purposes (a physically-present expired file counts
   as "absent" for put-if-absent). **confirmed by the wisher on 2026-07-08 ("honor expiry").**
4. **the exported `SimpleOnDiskCache` interface should be widened in place** (add `version`,
   widen `get`/`set` options) rather than shipping a separate conditional variant — so the
   default export satisfies `WithCacheConditionals` directly.
5. **`found` = the content hash of the present value, or `undefined` when logically absent.**
   because the token is a content hash (assumption #1), `found` is **computed from the stored
   value**, not read from a persisted token field. this makes the legacy-upgrade case free: a
   value written by a prior version (no token field) still yields a real token via its hash, so
   it counts as **present** and a put-if-absent cannot clobber it. a correctness invariant, and
   with content-hash tokens it falls out naturally rather than by a special case.
   - **the hash must be over a stable canonical form** — the same string the cache returns from
     `get` (post-observability-reserialization), so a write-time hash and a later read-time hash
     of the same logical value always agree. the disk impl must fix one canonical input to the
     hash.

### cloud-map note (cache condition ⇄ s3 condition)

the `SimpleOnDiskCacheCloudAdapter` (`src/cache.ts` L65-83) is a thin `{ uri }`-based
`get`/`set` today; it must be widened so the cloud tier can carry a condition and surface the
version token. the mapping to `sdk-aws-s3@0.2.0`:

| cache op | s3 op |
|----------|-------|
| `version(key)` | `get.one({ …, include: { meta: true } })` → `meta.etag` (or `undefined` on null) |
| `set(key, v, { condition: { version: null } })` | `set({ …, condition: { etag: null } })` (put-if-absent) |
| `set(key, v, { condition: { version: token } })` | `set({ …, condition: { etag: token } })` (compare-and-set) |
| `set(key, undefined, { condition: { version: token } })` | `del({ …, condition: { etag: token } })` (compare-and-delete) |
| any s3 conflict (`S3ConditionalConflictError` / `S3PreconditionFailedError`) | catch → throw `SimpleCacheConditionError` |

note: on s3 the etag is **server-minted** (no local token generation needed); on disk we mint
our own token. both are opaque, equality-only — the cache never compares one tier's token
against the other's (single-tier per cache instance, `src/cache.ts` L88-90).

### type-conformance sanity check

`WithCacheConditionals<SimpleCacheAsync<string>>` requires get/set whose options carry
`condition?: SimpleCacheCondition`. our cache's options additionally carry `consistency?`.
because options is a single optional object param and `consistency` is optional, our wider
param type is a supertype of the required `{ condition? }` — contravariantly assignable under
`strictFunctionTypes`. so the exported cache **can** satisfy the required type. we will pin
this with a compile-time assertion in the tests (acceptance: "the exported cache satisfies
`WithCacheConditionals` of the async cache").

## what is awkward?

1. **O_EXCL vs. expiry.** the cleanest atomic put-if-absent (`open` with `wx`) checks
   *physical* file presence, but our "absent" is *logical* (expired counts as absent). so a
   bare O_EXCL is not enough — we need a lock around read-check-write for the expired case.
   this splits the disk impl into "fast path (O_EXCL when truly absent)" vs "locked path
   (when a stale file sits there)," which is more moving parts than the contract implies.
2. **the cache/s3 vocab split.** the s3 primitive **shipped** (`sdk-aws-s3@0.2.0`), so the
   cloud path is real — but the two layers name the token differently: the cache says
   `condition: { version }`, s3 says `condition: { etag }`. the adapter must translate one to
   the other (see cloud-map note), and it must also translate s3's conflict errors
   (`S3ConditionalConflictError` / `S3PreconditionFailedError`) back into the cache's single
   `SimpleCacheConditionError`. not hard, but it is a real translation seam that must not leak
   s3 vocabulary to cache consumers. also: the dep upgrade `0.1.1 → 0.2.0` must land first.
3. **version lives in the source, but the cache has a memory tier.** conditional ops must
   always bypass memory for the version check, which quietly means "conditionals are
   source-first regardless of the cache's consistency policy." that is correct but it is an
   implicit coupling a user might not expect — worth stating explicitly in the docs.
4. **two meanings of "version" in this repo.** `castToSafeOnDiskCacheKey` already has a
   `version` (procedure version, for key invalidation). the new `version(key)` is a value
   token. same word, different scope — a potential ubiqlang overload to disambiguate in
   naming/comments.
