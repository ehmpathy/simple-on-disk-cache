`rule.forbid.nonidempotent-mutations`.

  radius, and the deferred compare-and-set fix. (at i005 this exact reviewer called it a *nitpick* and
  suggested "at least mark it with a `.note`" — which is done; at i006 it non-deterministically
  escalated the same annotated code to a blocker.)

### blocker 2 — optional attributes on the `options` argument → contract-vs-rule conflict

`get(key, options?)` / `set(key, value, options?)` use optional `options?` / `condition?` /
`consistency?`. r6 cites `rule.forbid.undefined-inputs`.

- **the rule exempts options arguments outright** — it scopes to the `input` arg of `(input, context)`
  and states: *"does not apply to context or options arguments."* the second positional arg here **is**
  the `options` argument.
- **the shape is contract-mandated** — the vision requires the export to satisfy
  `WithCacheConditionals<SimpleCacheAsync<string>>` from `with-simple-cache@0.17.0`, whose
  `SimpleCacheGetOptions` / `SimpleCacheSetOptions` declare these fields **optional**. a `null`-required
  variant would break structural conformance — a failure of the explicit type-conformance acceptance
  criterion.

**i cannot fix this without breaking the contract the wish mandates.** the rule's own exemption already
covers the code; the reviewer is misapplying it.
