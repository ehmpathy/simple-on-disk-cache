wish =

default to correct, opt-in to fast. flip the polarity of the in-process memory layer.

today `.get` is memory-first by default (`getWithMemory` in `src/cache.ts`): once a key is warm in memory, this process keeps its own copy and returns it even after another process/machine overwrites the source store. that makes the unsafe read the default and hides cross-process overwrites.

instead:

- default = source-first (correct). `.get` reads the source store (local disk / cloud) so it always reflects the latest write, cross-process overwrites included.
- opt-in = memory-first (fast). callers who know their cache is single-writer (or write-once / immutable) opt into the ~15ms→ns perf win explicitly.

why:

- pit-of-success. the safe, non-surprise behavior should be the default. a silently-stale read is a "wait, what?" moment; the person bitten is the one who never read the memory-layer internals. correctness should not require knowledge of an internal optimization.
- the perf win is real but situational. the memory layer erases ~15ms disk / hundreds of ms cloud per hot-key read. that is a genuine win — but only safe when one writer owns the store. it should be a conscious opt-in, not a silent default.
- the axis is memory-first vs source-first, not local vs cloud. the staleness bites the local-disk backend too (two processes on one box that share a dir, or a shared mount). cloud only makes it more visible.

desired shape:

```ts
// default: source-first (correct)
const cache = createCache({ directory });

// opt-in: memory-first (fast), for single-writer / write-once usecases
const cache = createCache({ directory, consistency: 'memory-first' });

// per-read override on a memory-first cache: force a source read
await cache.get(key, { consistency: 'source-first' });
```

- default `consistency: 'source-first'` — read source every time; memory may still be write-through so an explicit memory-first cache stays warm, but a source-first read never short-circuits on it.
- opt-in `consistency: 'memory-first'` — current behavior (memory hit short-circuits).
- `get(key, { consistency: 'source-first' })` — one-off source read even on a memory-first cache; the per-read value overrides the cache-wide default. covers lock/verify: write a token, wait, re-read source to confirm last writer.

acceptance:

- default is correct. on a default cache: `set(key, 'A')`, then a concurrent out-of-process overwrite to `'B'`, then `get(key)` returns `'B'`.
- opt-in is fast + unchanged. on `consistency: 'memory-first'`: after `set(key, 'A')`, a `get(key)` returns memory-cached `'A'` even after an out-of-process overwrite to `'B'` (current behavior preserved).
- per-read override works. on a memory-first cache, `get(key, { consistency: 'source-first' })` returns `'B'` after the overwrite, then leaves memory warm with `'B'` (or is documented as `memory: 'skip'`).
- both paths work for `local` and `cloud` directories.

out of scope:

- lossless high-concurrency valid-keys writes (the `TODO` in `src/cache.ts` around the valid-keys file) — separate concern.
- distributed cache invalidation / pub-sub — this wish is about read polarity, not automatic cross-machine coherence.

migration note:

this flips the default. per the wisher (2026-07-03), release as a **minor bump**: no caller receives a *wrong* value from the upgrade — single-process callers see only a latency change (memory always matched source anyway), and multi-writer callers see *corrected* values (the fix). document that speed-sensitive single-writer callers should set `consistency: 'memory-first'` to restore the old fast path.

context: found in the build of `with-simple-mutex` (ahbode/svc-notifications), where the settle-then-verify election had to re-read the source and the memory layer defeated it — which forced a drop down to sdk-aws-s3. supersedes ehmpathy/simple-on-disk-cache#40 — this correct-by-default model covers #40's per-read override need (now expressed as `consistency: 'source-first'` at read level).
