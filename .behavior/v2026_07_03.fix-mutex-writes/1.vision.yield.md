# vision: default to correct, opt-in to fast

## the outcome world

### day-in-the-life

before: a developer reaches for `simple-on-disk-cache` to coordinate state across two processes (a lock, an election, a shared flag on s3). they `set` a value, wait, and `get` it back to confirm — and the `get` returns their own stale copy from a hidden in-process memory layer. the cross-process overwrite is invisible. they lose hours, then discover the memory layer via a read of the source, then drop down to raw `sdk-aws-s3` to escape it. (this literally happened during a build of `with-simple-mutex` in ahbode/svc-notifications.)

after: the *next* developer with the same need uses `simple-on-disk-cache` as-is. `get` reads the source store and reflects the latest write — cross-process overwrites included. the lock/verify pattern just works, no bypass to `sdk-aws-s3`. the developer who *wants* the speed of the memory layer opts into it deliberately, with full knowledge of its single-writer assumption.

> **honesty note (self-review r2):** the original driver (`with-simple-mutex`) is **already unblocked** — it escaped via `sdk-aws-s3` directly. this vision does not rescue a currently-stuck caller; it is an **institutional fix** that stops the *next* caller from the same footgun and removes the need for that bypass going forward. the authority for the breaking default-flip is the human's explicit "default to correct" this session, not a live outage.

### before/after contrast

| aspect | before | after |
|--------|--------|-------|
| default `.get` | memory-first (fast, can be stale) | source-first (correct, always fresh) |
| cross-process overwrite visible? | no (until in-memory TTL expires) | yes (immediately) |
| who pays the surprise cost | correctness-sensitive caller who never read internals | nobody — the default is safe |
| speed opt-in | none (always on) | `consistency: 'memory-first'` |
| memory layer | silent, mandatory | explicit, optional |
| per-read override | none | `get(key, { consistency: 'source-first' })` — **in scope** (wisher-authorized) |

### aha moment

the developer writes a token to s3, waits, reads it back through the cache — and sees the *other* process's token, exactly as the source store holds it. the cache stops its lie about the world. "oh — it reads the actual store now, not its own memory."

## user experience

### usecases & goals

| persona | goal | mode they reach for |
|---------|------|---------------------|
| lock/election author | read-your-neighbor's-writes: confirm who last wrote the source | default (source-first) |
| single-writer read-heavy service | collapse N repeated reads of a hot key into 1 backend hit | opt-in `memory-first` |
| mostly-fast, occasionally-strict service | fast by default, but force a source-first read at a critical checkpoint | `memory-first` cache + `get(key, { consistency: 'source-first' })` — **in scope** |

**scope note (updated by wisher, 2026-07-03):** self-review r1 initially proposed to defer the per-read override (wet-over-dry — the lock/election usecase is served by the source-first default alone, and no mixed-mode caller was yet proven). the **wisher overrode**: it is **in scope**. phase-1 delivers all three — source-first default, `memory-first` opt-in, and the per-read `consistency` override.

### contract inputs & outputs

**construction — new optional `consistency`:**

```ts
// default: source-first (correct)
const cache = createCache({ directory: { cloud: { path, via } } });

// opt-in: memory-first (fast), for single-writer / write-once usecases
const cache = createCache({
  directory: { local: { path: './cache' } },
  consistency: 'memory-first',
});
```

**read — per-read `consistency` override (in scope, wisher-authorized):**

```ts
// on a memory-first cache, force one read past memory to the source
const latest = await cache.get(key, { consistency: 'source-first' });
```

**vocabulary decision (wisher, 2026-07-03):** the per-read override reuses the **same `consistency` term** as the construction option, rather than a separate `fresh: true` boolean. reasons (per `rule.require.ubiqlang`):
- **one canonical term.** `fresh` would be a second synonym for "source-first / bypass memory" — the exact overload ubiqlang forbids.
- **override symmetry.** the read-level `consistency` *overrides* the cache-level `consistency` default, like CSS specificity (narrower scope wins). same word, override relationship.
- **extensible.** if bounded-staleness becomes a mode, the per-read override names it too (`consistency: 'bounded'`); a boolean cannot express three states.

**scope decision (wisher, 2026-07-03):** self-review r1 initially proposed to defer this per-read override to phase 2 (wet-over-dry — no proven mixed-mode caller). the wisher overrode: it is **in scope**. it gives a memory-first cache a per-read way to force one read past memory to the source — useful for a strict checkpoint (lock/verify) inside an otherwise fast cache.

the `set` and `keys` signatures stay unchanged. the default `get(key)` signature stays unchanged; the `options` arg is additive (`consistency?: 'source-first' | 'memory-first'`).

### timeline (lock/verify, the usecase that drives this)

1. process A `set('lock', tokenA)` — writes to s3
2. process B `set('lock', tokenB)` — overwrites s3
3. process A `get('lock')` — **after**: returns `tokenB` (source truth); **before**: returned `tokenA` (own memory)
4. process A sees it is no longer the last writer → backs off. correct election.

## mental model

### how users would describe this to a friend

"the disk cache used to hand you back your own last write from memory, even if someone else changed the file. now it actually re-reads the file unless you tell it to go fast."

### analogies

- **before**: a sticky note on your monitor. you wrote "standup at 3", someone moved it to 4 on the shared calendar, but you keep reading your sticky note.
- **after**: you check the shared calendar by default. you *may* keep a sticky note (memory-first) when you know only you edit the calendar.

### terms

| what we call it | what users might call it |
|-----------------|--------------------------|
| source-first | fresh reads, consistent reads, "read the actual store" |
| memory-first | fast reads, cached reads, "the old behavior" |
| per-read `consistency: 'source-first'` | bypass cache, force refresh, no-cache read |
| the memory layer | the in-process cache, the hot cache |

note: users will likely say "consistent vs cached". we say "source-first vs memory-first" because the axis is *which layer answers the read*, not local-vs-cloud — the staleness bites the local-disk backend too.

## evaluation

### how well does it solve the goals?

| goal | solved? |
|------|---------|
| `.get` reflects cross-process overwrites by default | ✅ source-first default |
| preserve the perf win for those who want it | ✅ `memory-first` opt-in |
| targeted source-first read without a rebuild of the cache | ✅ `get(key, { consistency: 'source-first' })` — in scope (wisher-authorized) |
| pit-of-success (safe default, loud opt-in) | ✅ unsafe path now demands an explicit flag |

### pros

- correct by default — nobody re-discovers this defect the hard way
- perf win still fully available, just named and opt-in
- lock/verify patterns can use this package directly instead of a drop to `sdk-aws-s3`
- the name teaches the tradeoff (source vs memory) at the call site

### cons

- **the flip changes the default** — but the wisher decided **not to worry about backwards compat** (2026-07-03), shipped as a **minor bump**. this is well-justified: no caller receives a *wrong* value from the upgrade — single-process callers (the common case) see only a latency change (memory always matched source, A8), and multi-writer callers see *corrected* values (the fix). the only regression is speed for memory-first-reliant callers, restored with one line (`consistency: 'memory-first'`).
- source-first default is slower for read-heavy single-writer callers who do not yet know to opt in — but slow-and-correct beats fast-and-wrong as a default.
- **the source-first cost is two backend reads per `get`, not one** (self-review r2 / A4): the valid-key-tracked disk getter reads the reserved valid-keys record *then* the value (`src/cache.ts:406-415`). on cloud/S3 that is two round-trips. this makes the `memory-first` opt-in *more* valuable, not less — the win it buys (both reads → zero on a hit) is bigger than a naive "one ~15ms read" frame implies.

### edgecases & pit of success

| edgecase | how the contract keeps users safe |
|----------|-----------------------------------|
| single-writer service upgrades and silently slows down | changelog notes the flip; opt-in is one line (`consistency: 'memory-first'`) — no wrong values, only slower reads until opt-in |
| memory-first cache needs one strict read | `get(key, { consistency: 'source-first' })` — force one read past memory to the source, no cache rebuild |
| invalidation (`set(key, undefined)`) | source-first reads it every time anyway; memory-first already falls through on falsy (the memory short-circuit fires only on truthy, `src/cache.ts:435`) |
| source-first cache never trusts stale memory | a source-first cache **skips memory entirely** on read and write — no dead write-through (the in-memory cache is per-`createCache` closure at `src/cache.ts:425`, so a source-first cache would never read back what it wrote) |

## open questions & assumptions

### assumptions

1. **[answered]** the perf win is worth a keep as an opt-in (not full removal of the memory layer). — grounded: the `feat(perf)` commit `f13539a` added it precisely to erase ~15ms disk reads.
2. **[moot]** most current callers are single-writer, so the default-flip's value-change blast radius is small. — this mattered only to size a migration note; the wisher decided not to worry about backwards compat (2026-07-03), so the survey is no longer needed. the A8 logic (no caller gets a *wrong* value) already covers the safety concern without a headcount.
3. **[answered]** construction-level `consistency` is the right home for the cache-wide policy (vs #40's per-read-only approach). — grounded in the `input, options` pattern (mechanic brief `rule.require.input-options-pattern`): cache-wide policy belongs at construction. the per-read `consistency` override (wisher-authorized, in scope) reuses the same term to override the default for one-off checkpoints; both ship in phase 1.
4. source-first *reads* alone suffice for the lock usecase — the "out of scope" valid-keys write race does **not** undermine it. — **verified** (self-review r2/r3, A7): a source-first `get` gates on the valid-keys list (`src/cache.ts:406-415`, returns `undefined` if the key is absent from valid-keys), so this had to be checked. traced: the valid-keys race only drops *different* keys written in parallel; the lock/verify usecase writes the **same** key from both processes, so both add `'lock'` and the value file's last-writer-wins gives the fresh token. the scope boundary holds for the core usecase. minor near-TTL metadata-mismatch edge flagged for the build stone's test matrix.

### questions for the wisher

- **[answered by wisher, 2026-07-03] name = `consistency`.** `consistency: 'source-first' | 'memory-first'` at construction, reused as the per-read override. chosen over a `fresh` boolean because it names the *why*, names both poles, extends to bounded-staleness, and keeps one vocabulary across both scopes.
- **[answered by wisher, 2026-07-03] release as a minor bump.** self-review A8 classified this as semver-major, but the wisher chose **minor**: no caller receives a *wrong* value from the upgrade — single-process callers see only a latency change (memory always matched source), and multi-writer callers see *corrected* values (the fix). the only regression is speed for memory-first-reliant callers, who restore it with one line (`consistency: 'memory-first'`). the migration note documents that.
- **[answered by wisher, 2026-07-03] supersede #40 — yes.** this correct-by-default model covers #40's per-read override need (now `consistency: 'source-first'` at read level). #40 to be closed/repointed to this behavior.
- **[answered by wisher, 2026-07-03] bounded-staleness — deferred.** "we can do this in the future if needed." not built now (wet-over-dry — no proven hot-loop caller). the door stays open: the `consistency` enum extends to a `'bounded'` value later without a contract break.
- ~~per-read override — defer? name it `fresh: true`?~~ — **[answered by wisher, 2026-07-03]**: include it in phase 1 (reverses self-review r1's proposed deferral), and name it via the **same `consistency` term** (`get(key, { consistency: 'source-first' })`), not a separate `fresh` boolean — one canonical term, override symmetry, extensible.
- ~~default write-through~~ — **[answered]** by self-review r1: a source-first cache skips memory entirely (per-instance memory is never read back). no question remains.

- **sdk-aws-s3**: already the cloud adapter path (`src/cache.integration.test.ts:3`). the usecase that drives this (`with-simple-mutex`) currently bypasses this package via `sdk-aws-s3` directly — this vision removes that need.

## what is awkward?

- **the default-flip was the one sharp edge — now settled.** it slows read-heavy single-writer callers on upgrade. the wisher resolved this: ship as a minor bump, do not worry about backwards compat (2026-07-03). the one-line opt-in (`consistency: 'memory-first'`) restores speed for anyone who needs it. no open concern remains.
- **one term, two scopes.** `consistency` appears at both construction (cache-wide default) and per-read (single-read override) — the wisher authorized both for phase 1 and chose to reuse the *same* term rather than a separate `fresh` boolean. no conflation risk: it is one vocabulary with an override relationship (per-read wins over per-cache, like CSS specificity). a `source-first` cache treats a per-read `consistency: 'source-first'` as a no-op — it already reads source every time.
- **write-through on a source-first cache was dead work** — self-review r1 caught this. the in-memory cache is a per-`createCache` closure (`src/cache.ts:425`), so a source-first cache would populate memory it never reads. **cut it**: source-first skips memory on read and write. no awkwardness remains.
