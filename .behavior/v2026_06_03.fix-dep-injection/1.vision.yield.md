# vision: remote adapter dependency injection

## the outcome world

### before

```ts
import { createCache } from 'simple-on-disk-cache';

// user is forced to use AWS S3 — no choice
const cache = createCache({
  directory: {
    s3: { bucket: 'my-bucket', prefix: 'cache/' }
  }
});

// even if they only want local fs, the @aws-sdk/client-s3 dep
// is bundled, adding ~2MB to node_modules and internal cycles
```

pain points:
- **heavy dependency** — `@aws-sdk/client-s3` pulls in `@aws-crypto`, `@smithy/*`, etc.
- **vendor lock-in** — users on GCS, Azure Blob, R2, or custom storage can't use the package
- **dpdm exclusions** — the sdk has internal cyclical deps we must exclude in `.dpdmrc.yaml`
- **"simple" isn't simple** — a cache library shouldn't dictate cloud provider

### after

```ts
import { sdkAwsS3 } from 'sdk-aws-s3';
import { createCache } from 'simple-on-disk-cache';

// user passes SDK + URI — cache handles the rest
const cache = createCache({
  directory: {
    cloud: { path: 's3://my-bucket/cache/', via: sdkAwsS3 }
  }
});
```

benefits:
- **zero cloud deps** — `@aws-sdk/client-s3` removed from package.json
- **vendor agnostic** — S3, GCS, Azure, R2, local NFS, anything
- **truly simple** — the library does caching logic, user owns storage
- **clean dpdm** — no more exclusions for aws-sdk cycles

### the "aha" moment

> "wait, i can use this with cloudflare R2 now? and my bundle is 2MB lighter?"

the value clicks when users realize they're no longer paying the aws-sdk tax just to use a caching utility.

### usecases

| goal | how they achieve it |
|------|---------------------|
| local fs cache | `{ local: { path: './cache' } }` (unchanged) |
| S3 cache | `{ cloud: { path: 's3://bucket/prefix/', via: sdkAwsS3 } }` |
| GCS cache | `{ cloud: { path: 'gs://bucket/prefix/', via: sdkGcs } }` |
| Azure Blob cache | `{ cloud: { path: 'az://container/prefix/', via: sdkAzure } }` |
| R2 cache | `{ cloud: { path: 's3://bucket/prefix/', via: sdkAwsS3 } }` (S3-compat) |
| custom storage | `{ cloud: { path: 's3://...', via: sdkCustom } }` (SDK parses the URI) |

### contract inputs & outputs

**new type:**

```ts
/**
 * adapter for cloud storage backends
 *
 * .what = interface for storage SDKs that understand URI paths
 * .why = enables symmetric `{ path, via }` config for any cloud provider
 */
export type SimpleOnDiskCacheCloudAdapter = {
  /**
   * get a value by URI
   *
   * supports both:
   * - `get: { one: (input) => ... }` (namespace style, e.g., sdkAwsS3)
   * - `get: (input) => ...` (direct function style)
   *
   * @returns the value as a string, or null if not found (must NOT throw on not-found)
   */
  get:
    | { one: (input: { uri: string }) => Promise<string | null> }
    | ((input: { uri: string }) => Promise<string | null>);

  /**
   * set a value by URI
   */
  set: (input: { uri: string; body: string }) => Promise<void>;
};
```

**updated directory type:**

```ts
export type DirectoryToPersistTo =
  | { local: { path: string } }
  | { cloud: { path: string; via: SimpleOnDiskCacheCloudAdapter } };
```

### what it looks like to leverage them

**S3 (recommended — via sdk-aws-s3):**

```ts
import { sdkAwsS3 } from 'sdk-aws-s3';
import { createCache } from 'simple-on-disk-cache';

const cache = createCache({
  directory: {
    cloud: { path: 's3://my-cache-bucket/cache/', via: sdkAwsS3 }
  },
});
```

**GCS:**

```ts
import { sdkGcs } from 'sdk-gcs';  // not yet published
import { createCache } from 'simple-on-disk-cache';

const cache = createCache({
  directory: {
    cloud: { path: 'gs://my-cache-bucket/cache/', via: sdkGcs }
  },
});
```

**custom SDK (direct function style):**

```ts
import { S3Client, GetObjectCommand, PutObjectCommand } from '@aws-sdk/client-s3';
import { createCache, SimpleOnDiskCacheCloudAdapter } from 'simple-on-disk-cache';

const s3 = new S3Client({ region: 'us-east-1' });

const sdkCustom: SimpleOnDiskCacheCloudAdapter = {
  // direct function style (simpler)
  get: async ({ uri }) => {
    const { bucket, key } = parseS3Uri(uri);
    try {
      const { Body } = await s3.send(new GetObjectCommand({ Bucket: bucket, Key: key }));
      return Body ? await Body.transformToString('utf-8') : null;
    } catch (err: any) {
      if (err.name === 'NoSuchKey' || err.$metadata?.httpStatusCode === 404) return null;
      throw err;
    }
  },
  set: async ({ uri, body }) => {
    const { bucket, key } = parseS3Uri(uri);
    await s3.send(new PutObjectCommand({ Bucket: bucket, Key: key, Body: body }));
  },
};

// use it
const cache = createCache({
  directory: {
    cloud: { path: 's3://my-bucket/cache/', via: sdkCustom }
  },
});
```

## evaluation

### how well does it solve the goals?

| goal | solved? | notes |
|------|---------|-------|
| remove `@aws-sdk/client-s3` dep | ✅ yes | becomes user's optional dep |
| eliminate dpdm exclusions | ✅ yes | no more `@aws-sdk` in exclusions |
| support GCS/Azure/R2/etc | ✅ yes | any impl that satisfies `SimpleOnDiskCacheCloudAdapter` |
| local fs still works | ✅ yes | `{ local: {...} }` unchanged |
| backwards compat | ❌ no | hard cut, no deprecation |

### pros

- **lighter package** — no aws-sdk in deps
- **vendor agnostic** — any storage backend
- **simpler dpdm** — no exclusions needed
- **user control** — users configure their own client (region, auth, retries)
- **testable** — easy to mock `SimpleOnDiskCacheCloudAdapter` in tests

### cons

- **more user code** — users must write the adapter impl
- **migration effort** — extant `{ s3: {...} }` users must update
- **no batteries** — users might expect S3 to "just work"

### edgecases & pit of success

| edgecase | how we handle |
|----------|---------------|
| user returns `undefined` instead of `null` | type enforces `null` — typescript error |
| user's `get` throws on not-found | `sdk-aws-s3` handles; documented contract |
| user forgets to handle errors | their error propagates — cache doesn't swallow |
| concurrent writes | same as before — bottleneck for key file, user handles remote |

### pit of success for adapter correctness

the wish asked: "if they give us a get/set that doesn't do it" — what happens with a broken adapter?

**gotchas encoded in extant S3 client:**

| gotcha | extant behavior | if user misses it |
|--------|-----------------|-------------------|
| 404 detection | dual check: `err.name === 'NoSuchKey' OR err.$metadata.httpStatusCode === 404` | may not catch all 404 variants |
| 404 response | extant utils **throw** on 404; adapter must **return null** | adapter violates contract |
| string format | explicit `utf-8` in transformToString | may get wrong format |
| error context | wraps errors with bucket/key for debug | loses debug context |

**how we handle:**

1. **`sdk-aws-s3` recommended** — `sdk-aws-s3.get.one` returns `null` on 404, handles dual check, uses utf-8. users who use `sdk-aws-s3` get pit-of-success automatically.

2. **jsdoc contract** (required) — `SimpleOnDiskCacheCloudAdapter.get.one` jsdoc explicitly states: "must return null on not-found, not throw"

3. **runtime validation** (optional, for wisher) — could wrap user's adapter.get in try/catch at createCache time, detect throw-on-404 pattern via test call. adds complexity; may not be worth it.

**recommendation:** strong jsdoc + recommend `sdk-aws-s3`. users who want raw control can write adapters, but get clear contract. users who want easy path use `sdk-aws-s3` which handles gotchas.

## open questions & assumptions

### requirements from dream (already decided)

1. **recommend `sdk-aws-s3` for S3 users** — dream says "export current S3 utils as convenience (optional dep)".

   **resolution:** instead of export from this package, recommend separate `sdk-aws-s3` package. users pass SDK directly with URI:
   ```ts
   cloud: { path: 's3://bucket/prefix/', via: sdkAwsS3 }
   ```
   `sdk-aws-s3.get.one({ uri })` returns `null` on 404 and handles dual check.

2. **SimpleOnDiskCacheCloudAdapter only needs get.one/set** — SDK must provide `{ get: { one }, set }` that speak URI. deletion is encoded in the JSON metadata (`expiresAtMse: 0`), not as a separate operation. the SDK is a dumb key-value store; cache semantics live in the cache layer.

### assumptions

**explicit assumptions:**
1. **users can write adapters** — assumes users are comfortable with 2-4 lines of adapter code (mitigated by `sdk-aws-s3` recommendation)
2. **type safety is enough** — assumes `SimpleOnDiskCacheCloudAdapter` type catches most mistakes (TypeScript users; JS users have less safety)
3. **breaking change is acceptable** — wisher confirmed: removes S3 support, no deprecation period, stays v1

**inherited from extant design (not changed by this vision):**
4. **string values only** — extant cache stores strings; binary data must be base64 encoded
5. **key-value is sufficient** — adapter only needs get/set; key enumeration lives in cache layer via `RESERVED_CACHE_KEY_FOR_VALID_KEYS`
6. **errors propagate** — adapter errors reach caller; no built-in retry or wrap logic

1. ~~**deprecation timeline?**~~ — **answered:** breaking change, no deprecation, stays v1

## what is awkward?

### boilerplate comparison

| before | after |
|--------|-------|
| `{ s3: { bucket, prefix } }` | `{ cloud: { path: 's3://bucket/prefix/', via: sdkAwsS3 } }` |

nearly identical ergonomics. `via: sdkAwsS3` replaces implicit S3 dependence.

### null semantics

- `SimpleOnDiskCacheCloudAdapter.get.one` returns `null` for not-found
- `cache.get` returns `undefined` for not-found (cache converts `null` → `undefined`)

this aligns with extant behavior. consistent semantics throughout.

### no `get.all` required in SimpleOnDiskCacheCloudAdapter

the cache tracks valid keys internally via `RESERVED_CACHE_KEY_FOR_VALID_KEYS`. the adapter doesn't need `list()`.

this might surprise users who expect the adapter to handle key enumeration.

**mitigation:** document that key tracking is handled by the cache, not the adapter.
